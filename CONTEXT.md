## 배경

슬로우베드(Fursys 그룹, SCM 담당) 27년 성수기(1-3월) 공급망 운영 준비를 위해
26.10월~27.4월 매출을 예측하는 작업 중입니다. 영업 측 27년 사업계획이 아직 없어서
SCM이 자체적으로 시나리오 기반 예측 모델을 만들었고, 이를 유관부서 팀장들이
공동으로 가정값을 조정할 수 있는 웹 툴로 만들어뒀습니다. 이 프로젝트 폴더
(`forecast-collab-tool/`)가 그 결과물입니다.

## 데이터 배경

- 27.1~4월은 26년 동월 실적 YoY(1월 +17.8%, 2월 +5.8%, 3월 +16.1%, 4월 +7.0%)를

- 원본 Excel 모델: `27년_성수기_매출예측_시나리오모델.xlsx` (참고용, 로직의 원본 소스)

## 예측 로직 (forecast.js가 단일 소스)

보수/기본/공격 3-시나리오 + A~D 입력블록 구조는 **완전히 폐기**되었고, 월별로
"통계모델 예측 성장률 → 팀장 협의로 최종 확정" 하는 **단일 트랙**으로 최종 확정되었습니다
(회의에서 "이 달 성장률 몇 %로 합의했다"를 근거와 함께 한 번에 기록하는 형태).
시나리오조정값·특이요인효과(가격인상/신제품/신규점) 개념도 함께 제거됨.

각 예측월 = 전년동월 실적 × (1 + 최종협의성장률 또는 통계모델예측성장률)

- **통계모델 예측 성장률**(`reference_growth`, 읽기전용 시드값): 26.10~12월은 26.7~9월 평균
  성장률(9.0%) 대체 추정, 27.1~4월은 26년 동월 실적 YoY 그대로 준용 — 산출 근거 문구
  (`reference_rationale`)도 함께 시드되어 카드에 노출됨
- **매장(유통) 변동**(`store_change`) / **신제품 출시**(`new_product`): 팀장 누구나 자유서술로
  기록하는 텍스트 2칸 (기존 "영업환경" 1칸에서 분리됨 — 프로모션은 상시로 도는 거라 기록
  의미가 없고, 매장 증감/신제품 출시만 실질적으로 의미있는 변수라는 판단). 둘 다 계산에는
  사용 안 함(참고 정보)
- **최종 협의 성장률**(`final_growth`): 회의에서 합의된 값, `NULL`이면 "미확정" — 이 경우
  통계모델 예측 성장률로 잠정 계산
- **OEM/ODM가산은 이번 모델에서 완전히 제외** — 아파트멘터리 OEM/ODM 관련 조건이 아직
  미확정이라 있으면 혼란만 준다는 판단. 조건 확정되면 별도로 다시 반영 검토 (DB 컬럼도
  삭제됨, `forecast.js` 계산에서도 빠짐)
- **월별 카드 구성 순서**(위→아래): 통계모델 예측 성장률(하한/중심/상한 3버튼 토글, 미확정 월만) →
  매장(유통) 변동(자유서술) → 신제품 출시(자유서술) → 최종 협의 성장률(입력) → 예상 매출(계산값)
- **통계모델 예측 성장률 = 하한/중심/상한 3버튼 토글** (2026-07-03, Holt-Winters 갱신과 함께 도입):
  중심값은 새로 저장하지 않고 서버 `assumptions[ym].reference_growth`(DB 값)를 그대로 사용 —
  하한/상한만 `public/app.js`의 `REFERENCE_GROWTH_RANGE`에 정적 데이터로 하드코딩(Holt-Winters
  90% 예측구간, DB 스키마 변경 없음). 버튼 클릭 시 그 아래 "예상 매출" 값이 로컬에서만
  즉시 재계산되어 바뀌지만(`priorValue × (1+선택성장률)`), 서버 저장/`final_growth`엔 전혀
  영향 없는 순수 프론트엔드 미리보기 — 폴링/재렌더 시 "중심" 기본값으로 리셋되어도 무방하다는
  전제로 별도 영구 저장 안 함. `final_growth`가 이미 확정된 월은 토글이 의미가 없어 숨기고
  기존처럼 참고용 단일값만 표시
- 대외변수 참고자료 (소비심리지수, 기준금리, 아파트 입주물량, 매트리스 시장규모, 가구업계 실적) —
  DB에 없고 `public/app.js`의 `EXT_VARS`에 하드코딩. **월 카드 안에는 없고, 좌측 컬럼의 공통
  섹션 1곳에만 노출** (5개 지표가 월별로 동일한 값이라 카드마다 반복 노출하지 않음)

## 지금까지 만든 것 (현재 상태)

- **스택**: Node.js + Express, DB는 Postgres(`pg`) — 사내 Supabase(Self-hosted) 인스턴스에
  연결 (SQLite 아님 — 과거 SQLite로 갔던 결정은 폐기됨, 아래 "배포 관련 결론" 참고)
- **프론트**: 순수 HTML/CSS/JS, 빌드 툴 없음, Chart.js·`@supabase/supabase-js` 모두 CDN
- **편집자 이름 게이트 없음, 변경 이력 추적 안 함**: 실시간 협업 툴이라 누가 바꿨는지 기록할
  필요가 없다는 결론. "내 이름" 입력칸, `changed_by` 파라미터, `history` 테이블/변경이력
  패널을 전부 제거함. 값이 바뀌면 즉시 저장되고, 저장 성공/실패 토스트만 뜸 (누가 바꿨는지는

- **동기화**: 팀장들이 월 카드에서 값을 수정하면 즉시 저장 → Supabase Realtime
  (`postgres_changes`, `month_assumptions` 테이블 구독)으로 다른 사용자 화면에도 즉시 반영,
  웹소켓 끊김 대비 60초 폴백 폴링 병행

  - **왼쪽**: 매출 추이 차트 + 대외변수 참고 (하나의 카드 프레임, 내용이 넘치면 카드 프레임

## 프로덕션 DB 변경 작업 프로세스 (2026-07-03 확정, 회사 보안 정책)

**`DATABASE_URL`은 어떤 경우에도 Claude Code 세션에 입력하지 않는다.** 회사 보안 정책상
AI 세션이 프로덕션 DB 크리덴셜을 직접 보유·접속하는 것 자체가 금지됨. 이전에 로컬 검증
목적으로 `pg-mem`(인메모리 Postgres 에뮬레이터)을 썼던 것은 실제 프로덕션 크리덴셜이
아니라 가짜 로컬 DB였으므로 이 정책과 무관함 — 실제 프로덕션 DB 접속 자체가 금지 대상.

DB 변경(조회 포함 여부와 무관하게 UPDATE/DELETE 등 변경이 필요한 경우)이 필요하면:
1. Claude Code는 `sql/` 폴더에 SQL 스크립트 파일만 준비한다 (실행하지 않음)
2. 변경 전 상태를 먼저 확인할 수 있는 순수 조회(SELECT) 스크립트를 별도 파일로 함께 제공
3. 변경 스크립트는 `BEGIN ... COMMIT` 트랜잭션으로 감싸고, 변경 전/후 상태를 스크립트 안에서
   SELECT로 비교할 수 있게 구성, 건드리면 안 되는 컬럼(`final_growth` 등 팀 협의값)은 SET
   절에서 명시적으로 제외
4. 실제 실행은 사용자가 Supabase SQL Editor에서 직접 하거나 IT팀에 전달해서 처리한다

예시: `sql/check_final_growth.sql`(조회 전용), `sql/update_reference_growth_holtwinters.sql`
(트랜잭션 UPDATE) — 둘 다 민감정보 없는 순수 SQL이라 git에 커밋됨.
