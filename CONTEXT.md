# 27년 성수기 매출예측 협업 보드 — Claude Code 인계 프롬프트

아래 내용을 Claude Code 첫 메시지로 그대로 붙여넣어 사용하세요.
(이 파일 자체를 프로젝트 루트에 두고 "CONTEXT.md 읽고 시작해줘"라고 해도 됩니다.)

## 배경

슬로우베드(Fursys 그룹, SCM 담당) 27년 성수기(1-3월) 공급망 운영 준비를 위해
26.10월~27.4월 매출을 예측하는 작업 중입니다. 영업 측 27년 사업계획이 아직 없어서
SCM이 자체적으로 시나리오 기반 예측 모델을 만들었고, 이를 유관부서 팀장들이
공동으로 가정값을 조정할 수 있는 웹 툴로 만들어뒀습니다. 이 프로젝트 폴더
(`forecast-collab-tool/`)가 그 결과물입니다.

## 데이터 배경

- 원본: 25.1월~26.9월 월매출 실적 (백만원 단위), 26.7~9월은 영업협의 예상치
- 25년 1월부터만 데이터가 있어 26.10~12월은 전년동월 YoY 산출이 안 됨 →
  26.7~9월 평균 성장률(9.0%)로 대체 추정
- 27.1~4월은 26년 동월 실적 YoY(1월 +17.8%, 2월 +5.8%, 3월 +16.1%, 4월 +7.0%)를
  그대로 준용한 가정치 (다소 낙관적일 수 있음, 확정되면 갱신 필요)
- 원본 Excel 모델: `27년_성수기_매출예측_시나리오모델.xlsx` (참고용, 로직의 원본 소스)

## 예측 로직 (forecast.js가 단일 소스)

보수/기본/공격 3-시나리오 + A~D 입력블록 구조는 **완전히 폐기**되었고, 월별로
"로직 근거 성장률 → 팀장 협의로 최종 확정" 하는 **단일 트랙**으로 최종 확정되었습니다
(회의에서 "이 달 성장률 몇 %로 합의했다"를 근거와 함께 한 번에 기록하는 형태).
시나리오조정값·특이요인효과(가격인상/신제품/신규점) 개념도 함께 제거됨.

각 예측월 = 전년동월 실적 × (1 + 최종협의성장률 또는 로직근거성장률)

- **로직 근거 성장률**(`reference_growth`, 읽기전용 시드값): 26.10~12월은 26.7~9월 평균
  성장률(9.0%) 대체 추정, 27.1~4월은 26년 동월 실적 YoY 그대로 준용 — 산출 근거 문구
  (`reference_rationale`)도 함께 시드되어 카드에 노출됨
- **매장(유통) 변동**(`store_change`) / **신제품 출시**(`new_product`): 팀장 누구나 자유서술로
  기록하는 텍스트 2칸 (기존 "영업환경" 1칸에서 분리됨 — 프로모션은 상시로 도는 거라 기록
  의미가 없고, 매장 증감/신제품 출시만 실질적으로 의미있는 변수라는 판단). 둘 다 계산에는
  사용 안 함(참고 정보)
- **최종 협의 성장률**(`final_growth`): 회의에서 합의된 값, `NULL`이면 "미확정" — 이 경우
  로직 근거 성장률로 잠정 계산
- **OEM/ODM가산은 이번 모델에서 완전히 제외** — 아파트멘터리 OEM/ODM 관련 조건이 아직
  미확정이라 있으면 혼란만 준다는 판단. 조건 확정되면 별도로 다시 반영 검토 (DB 컬럼도
  삭제됨, `forecast.js` 계산에서도 빠짐)
- **월별 카드 구성 순서**(위→아래): 로직 근거 성장률(읽기전용) → 매장(유통) 변동(자유서술) →
  신제품 출시(자유서술) → 최종 협의 성장률(입력) → 예상 매출(계산값)
- 대외변수 참고자료 (소비심리지수, 기준금리, 아파트 입주물량, 매트리스 시장규모, 가구업계 실적) —
  DB에 없고 `public/app.js`의 `EXT_VARS`에 하드코딩. **월 카드 안에는 없고, 좌측 컬럼의 공통
  섹션 1곳에만 노출** (5개 지표가 월별로 동일한 값이라 카드마다 반복 노출하지 않음)

## 지금까지 만든 것 (현재 상태)

- **스택**: Node.js + Express, DB는 Postgres(`pg`) — 사내 Supabase(Self-hosted) 인스턴스에
  연결 (SQLite 아님 — 과거 SQLite로 갔던 결정은 폐기됨, 아래 "배포 관련 결론" 참고)
- **프론트**: 순수 HTML/CSS/JS, 빌드 툴 없음, Chart.js·`@supabase/supabase-js` 모두 CDN
- **편집자 이름 게이트 없음, 변경 이력 추적 안 함**: 실시간 협업 툴이라 누가 바꿨는지 기록할
  필요가 없다는 결론. "내 이름" 입력칸, `changed_by` 파라미터, `history` 테이블/변경이력
  패널을 전부 제거함. 값이 바뀌면 즉시 저장되고, 저장 성공/실패 토스트만 뜸 (누가 바꿨는지는
  더 이상 어디에도 안 남음)
- **동기화**: 팀장들이 월 카드에서 값을 수정하면 즉시 저장 → Supabase Realtime
  (`postgres_changes`, `month_assumptions` 테이블 구독)으로 다른 사용자 화면에도 즉시 반영,
  웹소켓 끊김 대비 60초 폴백 폴링 병행
- **레이아웃**: 헤더 아래를 좌우 2컬럼(`grid-template-columns: 42% 1fr`)으로 분할.
  - **왼쪽**: 매출 추이 차트 + 대외변수 참고 (하나의 카드 프레임, 내용이 넘치면 카드 프레임
    안에서만 자체 스크롤)
  - **오른쪽**: 월별 카드 그리드 (역시 카드 프레임 안에서 자체 스크롤)
  - 좌우 모두 "바깥은 테두리/모서리만 담당, 안쪽 div가 스크롤 담당"으로 분리되어 있어서
    스크롤 중에도 모서리가 잘려 보이지 않음. 페이지 자체는 `body`가 뷰포트 높이에 고정된
    flex 셸이라 헤더 높이를 px로 추정해서 sticky/max-height를 계산하지 않음(추정치가 틀리면
    깨지는 문제를 근본적으로 없앰)
  - 최상위 컨테이너에 `max-width` 제한 없음(좌우 padding만 24px) — 넓은 화면에서 오른쪽
    카드가 3~4열까지 자연스럽게 늘어남
  - 900px 이하에서는 이 2컬럼/셸 구조가 전부 풀리고 기존처럼 세로 1컬럼(차트 → 대외변수 →
    카드)으로 자연스럽게 스크롤되는 페이지로 전환됨
- **차트** (25.1월 ~ 27.4월 매출 추이, 왼쪽 컬럼 상단): 매출=막대(왼쪽 축, 백만원 —
  실적/예상/확정예측은 진한 색, 잠정예측은 옅은 색) + YoY 성장률=꺾은선(오른쪽 보조 축, % —
  26.1월부터만 표시, 25년 구간은 전년 데이터 없어 비움, 26.10월부터는 매출·성장률 모두
  점선으로 바뀌어 실적/예측 구간 구분) 콤보 차트. 툴팁에 매출값+성장률 같이 표시. 차트
  헤더에 "금액은 공장도가 기준입니다" 캡션
- **데이터 모델**: `month_assumptions` 월 1행×7행 구조 (구 시나리오 기반 단일행
  `assumptions` 테이블과 `history` 테이블은 `db.js`의 `initDb()`가 자동으로 DROP,
  `oem_addback`/`updated_by`/`business_context` 컬럼도 삭제되고 `store_change`/`new_product`로
  교체됨 — 기존 배포 DB에 대해서도 방어적으로 DROP/ADD 처리)
- **검증 완료 (세션 중, mock pg로)**: 문법 체크 + `pg`를 메모리 mock으로 대체해 서버를 띄운
  뒤 seed된 로직근거성장률로 예측 계산, final_growth 설정/미확정 복귀, `changed_by` 없이 PUT
  저장, `GET /api/state`에 history/oem/business_context 필드가 안 나오고 store_change/
  new_product는 나오는지, 예상매출이 OEM 없이 순수 `전년동월실적×(1+성장률)`로 계산되는지,
  Infinity 등 비정상값 거부, 화이트리스트 밖 필드 수정 차단, 정적 파일 서빙까지 전부 확인.
  **실제 Postgres/Supabase 연결·Realtime 동작, 실제 브라우저 렌더링(레이아웃/차트 시각적
  결과)은 이 환경에 크리덴셜·Docker·헤드리스 브라우저가 없어 검증 불가** — 아래 "다음 단계"
  참고
- 파일 구조는 `README.md` 참고. `.env` 채운 뒤 `npm install && npm start` → `localhost:4747`

## 배포 관련 결론 (SQLite 결정은 폐기됨, 재논의 불필요라고 적어도 상황 바뀌면 다시 논의될 수
있으니 이 섹션 자체를 최신으로 유지할 것)

- **DB는 Postgres로 확정** — 사내 Supabase(Self-hosted) 인스턴스, 접속은
  `@supabase/supabase-js`가 아니라 `pg` 드라이버로 직접 연결
- **동기화**: Supabase Realtime(`postgres_changes`, `month_assumptions` 테이블만 구독 —
  `history` 테이블은 폐기되어 구독 대상 아님)으로 즉시 반영, 웹소켓 끊김 대비 60초 폴백
  폴링 병행
- **배포**: Coolify (git 레포 기반, `Dockerfile` 포함되어 있음)
- GitHub Pages는 여전히 배포 후보 아님 (서버가 필요해서)
- **git 저장소**: 오늘 로컬에 `git init` 완료, 커밋도 로컬에 되어 있음. **원격 저장소는
  아직 연결 안 됨** — Coolify에 연결하려면 원격 레포(사내 Git 서버든 GitHub든)부터 필요

### 미확정 (확인 필요)
- `DATABASE_URL` 크리덴셜의 권한 레벨 (Realtime publication/RLS 정책을 앱 크리덴셜로
  실행 가능한지, 아니면 별도 관리자 권한이 필요한지) — `sql/enable_realtime.sql` 참고
- Postgres 접속 시 SSL 필요 여부 (사내망 직접 연결인지 TLS 종단 프록시 경유인지)
- Coolify에 연결할 git 원격 저장소 (로컬 커밋까지만 되어 있고 원격은 아직 미정)
- 실제 Supabase 프로젝트에 스키마가 적용되어 있는지 (아직 미확인)
